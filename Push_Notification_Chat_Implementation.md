# Comprehensive Technical Guide: Push Notifications & Real-Time Chat Module

This guide provides a detailed, step-by-step blueprint for implementing a robust Push Notification system combined with a WhatsApp/iMessage-style Chat Module. It is designed to be ingested by an AI or a developer to cleanly replicate this architecture in any modern web application (e.g., Next.js, React, Flutter) connected to a backend (like Supabase, Firebase, or a custom Node.js server).

---

## 1. High-Level Architecture
The system relies on an event-driven architecture decoupled into four layers:

1. **Frontend (UI & Service Worker)**: Manages local chat state (Optimistic UI), requests browser permission, registers the Service Worker, and stores the Push Subscription (endpoint + keys).
2. **Database (State & Sync)**: Stores user subscriptions, Chat/Comments history, and an In-App Notification inbox.
3. **Trigger Abstraction (Client-side wrappers)**: A centralized module that observes actions (e.g., "New Comment Sent") and triggers the backend notification service.
4. **Backend Push Service (Edge Function / Serverless)**: A secure, isolated function that holds the VAPID Private Key. It constructs the payload, saves the notification to the DB, and dispatches it via the `web-push` protocol to all active devices of the target user.

## 2. Database Schema Specification

To support this architecture, implement the following relational schema:

### A. Push Subscriptions Table
Stores active browser/device push endpoints.
```sql
CREATE TABLE push_subscriptions (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    endpoint TEXT NOT NULL UNIQUE,
    p256dh TEXT NOT NULL,
    auth TEXT NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
-- Unique constraint on (user_id, endpoint) to prevent duplicates per device
```

### B. In-App Notifications Table
Serves as an inbox for users who missed the push or have push disabled.
```sql
CREATE TABLE notifications (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    entity_id UUID, -- E.g., request_id or chat_thread_id
    type VARCHAR(50), -- 'comment', 'status_change', 'alert'
    title TEXT NOT NULL,
    message TEXT NOT NULL,
    is_read BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

### C. Chat/Comments Table
Stores the actual chat history.
```sql
CREATE TABLE request_comments (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    reference_id UUID NOT NULL, -- The ticket/request the chat belongs to
    author_id UUID REFERENCES users(id),
    body TEXT NOT NULL,
    is_internal BOOLEAN DEFAULT FALSE, -- For "Admin/Staff Only" hidden notes
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 3. Frontend: Service Worker & Registration

### Step 3.1: The Service Worker (`sw.js`)
Create a file named `sw.js` in your `public/` directory. This runs in the background to receive push events even when the app is closed.

```javascript
self.addEventListener('push', function(event) {
    const data = event.data ? event.data.json() : {};
    
    const title = data.title || 'New Notification';
    const options = {
        body: data.body,
        icon: data.icon || '/icon-192.png',
        badge: data.badge || '/badge-72.png',
        data: { url: data.url || '/' },
        vibrate: [100, 50, 100],
    };

    event.waitUntil(self.registration.showNotification(title, options));
});

self.addEventListener('notificationclick', function(event) {
    event.notification.close();
    if (event.notification.data.url) {
        event.waitUntil(clients.openWindow(event.notification.data.url));
    }
});
```

### Step 3.2: Push Registration Utility
Create a helper function to prompt the user and save their endpoint to your DB. Note: The VAPID Public Key can be safely exposed to the client.

```typescript
// utils/pushNotifications.ts
const VAPID_PUBLIC_KEY = 'YOUR_VAPID_PUBLIC_KEY'; // Base64 URL-safe

function urlBase64ToUint8Array(base64String: string): Uint8Array {
    const padding = '='.repeat((4 - (base64String.length % 4)) % 4);
    const base64 = (base64String + padding).replace(/-/g, '+').replace(/_/g, '/');
    const rawData = window.atob(base64);
    const outputArray = new Uint8Array(rawData.length);
    for (let i = 0; i < rawData.length; ++i) {
        outputArray[i] = rawData.charCodeAt(i);
    }
    return outputArray;
}

export async function subscribeToPush(userId: string) {
    // 1. Register SW
    const swReg = await navigator.serviceWorker.register('/sw.js');
    await navigator.serviceWorker.ready;

    // 2. Subscribe via PushManager
    const subscription = await swReg.pushManager.subscribe({
        userVisibleOnly: true,
        applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY)
    });

    const subJson = subscription.toJSON();

    // 3. Upsert to Database (Pseudocode)
    await db.from('push_subscriptions').upsert({
        user_id: userId,
        endpoint: subJson.endpoint,
        p256dh: subJson.keys.p256dh,
        auth: subJson.keys.auth
    }, { onConflict: 'user_id, endpoint' });
}
```

## 4. Backend: The Secure Push Sender (Edge Function)

The backend is strictly responsible for using the `web-push` library with the VAPID Private Key. Do **NOT** expose this key to the frontend.

### Implementation Logic (Node.js/Deno snippet):
```typescript
import webpush from 'web-push';

webpush.setVapidDetails(
    'mailto:admin@yourdomain.com',
    process.env.VAPID_PUBLIC_KEY,
    process.env.VAPID_PRIVATE_KEY
);

export async function handleSendPush(req, res) {
    const { user_id, title, body, url } = req.body;

    // 1. Store in In-App Notifications table
    await db.from('notifications').insert({
        user_id, title, message: body, url, type: 'push'
    });
    
    // (Optional) Cleanup Old Notifications to prevent DB bloat
    // DELETE from notifications WHERE user_id = X AND id NOT IN (Latest 50)

    // 2. Fetch all active subscriptions for the user
    const subscriptions = await db.from('push_subscriptions').select('*').eq('user_id', user_id);

    // 3. Dispatch Push sequentially or via Promise.all
    const payload = JSON.stringify({ title, body, url });

    for (const sub of subscriptions) {
        try {
            await webpush.sendNotification({
                endpoint: sub.endpoint,
                keys: { p256dh: sub.p256dh, auth: sub.auth }
            }, payload);
        } catch (error) {
            // CRITICAL: Cleanup Stale Subscriptions
            // If user uninstalls app or revokes permission, push service returns 404 or 410
            if (error.statusCode === 404 || error.statusCode === 410) {
                await db.from('push_subscriptions').delete().eq('id', sub.id);
            }
        }
    }

    return res.status(200).json({ success: true });
}
```

## 5. Abstraction: Notification Trigger Layer

To keep your UI components clean, abstract the API calls to your Edge Function into a trigger layer.

```typescript
// lib/notificationTrigger.ts

export async function sendNotification(payload: { user_id: string; title: string; body: string; url: string }) {
    await fetch('/api/send-push', { method: 'POST', body: JSON.stringify(payload) });
}

// Scenario 1: Admin replies to user
export async function notifyUserOfReply(userId: string, ticketId: string, replyPreview: string) {
    await sendNotification({
        user_id: userId,
        title: '💬 New Reply from Support',
        body: replyPreview,
        url: `/ticket/${ticketId}`
    });
}

// Scenario 2: User replies (Notify all admins)
export async function notifyAdminsOfReply(ticketId: string, replyPreview: string) {
    const admins = await db.from('users').select('id').eq('role', 'admin');
    for (const admin of admins) {
        await sendNotification({
            user_id: admin.id,
            title: '💬 User replied to ticket',
            body: replyPreview,
            url: `/admin/ticket/${ticketId}`
        });
    }
}
```

## 6. The Chat Module (Optimistic UI)

The Chat Module displays messages in a "Bubble" format. It uses an optimistic update pattern: the moment the user hits 'Send', the UI updates immediately, and the backend syncs asynchronously.

### Architecture Rules for Chat Bubble:
1. **Layout**: Use CSS Flexbox. 
   - `flex-row`: Align Left (e.g., The other party / User). Gray background.
   - `flex-row-reverse`: Align Right (e.g., the current logged-in user). Primary Brand Color background.
2. **Internal Flagging**: Allow Admins to toggle a boolean `is_internal`. If `is_internal = true`, render the bubble with a distinct color (e.g., Amber/Yellow) and ensure your API naturally filters these out when a regular user pulls comments.
3. **Scroll to Bottom**: Tie a React `useEffect` to the `comments` array length to ensure the `scrollRef.scrollTop = scrollRef.scrollHeight`.

### Functional Flow (Submit Event):
```typescript
const handleSendMessage = async (text: string, isInternal: boolean) => {
    // 1. Optimistically append to local state immediately
    const tempComment = { id: Date.now(), body: text, author_id: currentUser.id, is_internal: isInternal };
    setLocalComments(prev => [...prev, tempComment]);

    // 2. Insert into Database
    const newCommentDB = await insertCommentToDB(ticketId, text, isInternal);

    // 3. Trigger Push Notification Logic Based on Role
    if (currentUser.role === 'admin') {
         // Notify the creator of the ticket
         await notifyUserOfReply(ticketCreatorId, ticketId, text);
    } else {
         // Notify the support team
         await notifyAdminsOfReply(ticketId, text);
    }
}
```

## 7. Implementation Best Practices for Next-Gen AIs

When applying this blueprint, strictly adhere to the following techniques:

1. **Stale Endpoint Cleanup**: The most common point of failure in Web Push is a bloated database of dead endpoints. You **must** catch `410 Gone` and `404 Not Found` errors in the Push Service and actively `DELETE` them from the `push_subscriptions` table.
2. **De-coupled Notification Inbox**: Always write notifications to the DB first. Users may refuse browser notification permissions; the `notifications` table ensures they still get a red badge/bell icon inside the app.
3. **Data Hydration**: Pass variables like sender display names or ticket IDs dynamically into the Push Title/Body using formatting strings before sending to the push function.
4. **VAPID Security**: `VAPID_PUBLIC_KEY` goes into frontend `.env.public`, but `VAPID_PRIVATE_KEY` strictly goes into backend `.local.env` or server secrets. They only act as a cryptographic handshake to verify you are the legitimate sender to Google/Apple push servers.
5. **Optimistic Rendering**: Forms inside chat threads should not block or show a loading spinner over the whole component. Instantly clear the text box and append the comment bubble so the application feels native and instantaneous.
